Failure Modes & Recovery
Component / Failure
Impact
Automatic Recovery
Severity
CP Gateway session expires
No orders can be placed or queried. Pending trades stall at wait_for_fill.
Tickle λ detects auth failure → attempts reauthenticate × 3 → disables trading + SNS alert if all fail. Manual: restart CP Gateway container via SSM.
HIGH
CP Gateway container crashes
All IB API calls fail. Open positions cannot be monitored or closed.
Docker restart: unless-stopped restarts container automatically. Dead Man λ closes any stale positions after 15min heartbeat gap.
HIGH
Lambda function cold start delay
First invocation after idle period adds ~500ms–2s latency.
Step Functions timeout/retry margins absorb cold starts. Pre-warm not required at this throughput.
LOW
Step Functions execution failure
Trade lifecycle halts at failed state. Position may be open with no monitoring.
EmergencyClose state catches all unhandled errors and attempts MKT close. Dead Man λ covers residual orphans within 15min.
MED
Fargate task crash (dashboard-api)
Dashboard unavailable. No live price feed. REST API down. Trading continues.
ECS service restarts task automatically. ALB health check removes unhealthy targets. Trading unaffected — Fargate is read-only path for dashboard.
LOW
DynamoDB throttling
Lambda writes/reads fail. Heartbeats miss. risk_check may block on position count.
On-demand billing mode scales automatically. Lambda retries cover transient throttles. Dead Man λ covers prolonged outage.
LOW
Order fill timeout (60s)
Entry order not confirmed within 60s — likely rejected at exchange or stale.
WaitForFill state catches States.Timeout → CancelOrder state → IB DELETE order → OrderFailedLog. No position opened.
LOW
GTC stop order not cancelled after close
Resting stop fires next trading day → opens unwanted reverse position.
close_position λ always cancels GTC stop before MKT close (unconditional, swallows 404). portfolio_risk and dead_man both cancel stops before emergency close.
HIGH (fixed)
AWS region outage
All components unavailable. Open positions at IBKR have no monitoring.
No automatic recovery (single-region). Manual: log into IB web portal and close open positions. Dead Man λ in same region — also affected. Accept residual risk for cost.
HIGH (accepted)
Operational Runbooks
Trading disabled unexpectedly Alert
Check CloudWatch Logs: trade-engine-*/dead_man or portfolio_risk
Check DynamoDB ConfigTable: pk=trading_enabled — read reason field
If reason=dead_man_triggered: identify orphaned trade, verify IB position is flat
If reason=portfolio_risk_limit: review daily P&L, wait for next session
Re-enable: Dashboard → Resume Trading OR aws dynamodb put-item … pk=trading_enabled value=true reason=manual_resume
CP Gateway auth failure SNS Alert
Check CloudWatch Logs: tickle Lambda — look for reauth failure messages
SSM Session Manager → EC2 instance → cd /opt/ibgateway && docker-compose ps
If container running but session expired: docker-compose restart ibgateway
If container not running: docker-compose up -d
Wait 2 min for IB auth — Tickle λ auto-detects session alive and re-enables trading
Note: if dead_man/portfolio_risk fired during outage, manual re-enable required
Position stuck OPEN — no heartbeat Manual check
Dashboard → Positions tab — check last_heartbeat timestamp
If Dead Man hasn't fired yet: Dashboard → Close Position (manual)
Verify IB web portal: confirm position is actually flat after close
Check SF execution in console — may be in EmergencyClose state
Signal rejected unexpectedly Investigate
Check NinjaTrader Output window for rejection reason from OrchestratorClient
Check CloudWatch Logs: risk_check Lambda for rejection reason
Common causes: outside RTH · daily loss limit · position limit · trading_enabled=false
Dashboard → Risk tab shows current daily P&L and open position count
Emergency: close all positions immediately Manual
Dashboard → Close All Positions (stops SF executions, cancels stops, MKT close all)
If dashboard unavailable: IB web portal → Portfolio → close all manually
Verify all positions flat in IB before leaving terminal
Dashboard → Pause Trading to prevent new signals while investigating
Deployment Runbook
Full stack deployment (first time) ~20 min
cd infra && sam build
sam deploy --guided — fill in parameters (IB creds, phone, etc.)
Build & push dashboard-api Docker image to ECR
Upload dashboard-ui static files to S3 bucket (from CFN outputs)
Update OrchestratorClient.cs BaseUrl → ApiGatewayUrl output, recompile in NinjaTrader
Set TRADE_ENGINE_API_KEY Windows env var → restart NinjaTrader
Lambda code update ~2 min
sam build && sam deploy (non-interactive, uses samconfig.toml)
Lambdas update in-place — no downtime. In-flight SF executions use previous version until they complete.
Verify in CloudWatch Logs that new code is running
Dashboard-api Docker update ~5 min
docker build -t trade-engine-dashboard:latest ./dashboard-api
docker push {accountId}.dkr.ecr.{region}.amazonaws.com/trade-engine-dashboard-{stack}:latest
aws ecs update-service --force-new-deployment … — ECS rolling update
Deploy outside market hours — rolling restart briefly interrupts WebSocket connections
IB credentials rotation Out-of-hours
Update Secrets Manager secret: aws secretsmanager update-secret …
Restart CP Gateway container: docker-compose restart ibgateway
UserData re-fetches secret on container restart — new credentials picked up automatically
API key rotation ~3 min
Generate new key, update Secrets Manager
Update Windows env var TRADE_ENGINE_API_KEY → restart NinjaTrader
api_authorizer λ picks up new key on next cold start (or force new deployment)
