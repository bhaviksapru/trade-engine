Trade-offs
Cost Reduction Options

The defaults above already apply Fargate Spot and scheduled EC2 start/stop. The options below go further — each carries real operational consequences.

A
Single NAT
Gateway
⚠ Medium Risk
Save ~$14.22/month

Remove NAT GW in AZ-b and AZ-c. All private route tables point at AZ-a's gateway. If AZ-a's NAT fails, Lambda and Fargate in AZ-b/c lose internet egress — the tickle Lambda stops, and active trades risk orphaning until dead_man closes them. The EC2 CP Gateway itself is in AZ-a so its IBKR traffic is unaffected.

New total: ~$27.75/mo
B
NAT Instance
per AZ
⚠ Medium Risk
Save ~$18.85/month vs 3× GW

Replace each NAT Gateway with a t3.nano EC2 instance running Amazon Linux 2 with IP forwarding + iptables MASQUERADE. 3 instances × 157.5hrs = ~$2.46/mo. t3.nano baseline is ~32 Mbps — well above IBKR's single-contract data feed. Trade-off: manual patching, instance replacement on hardware failure, lower SLA than managed NAT GW.

New total: ~$21.12/mo
C
Single-AZ
Deployment
✕ High Risk
Save ~$14/month vs 3-AZ

Run everything in one AZ — one public subnet, one private subnet, one NAT gateway. No AZ redundancy: an AZ outage takes down EC2, Fargate, and Lambda simultaneously. Correct choice for development environments or if NinjaTrader and the system share the same physical machine (making AZ redundancy academic).

New total: ~$27/mo
